# Baita Bot Creation Guide — AI-Assisted Reference

This document provides complete instructions for programmatically creating bots via the Baita API. It is designed to be consumed by an AI assistant that helps users build automation bots through conversation.

## Overview

A Baita bot is a serverless automation workflow (similar to Zapier zaps). Each bot:
- Has a **trigger** (how it starts: webhook HTTP call or schedule)
- Has one or more **tasks** (sequential steps that execute logic)
- Chains data between tasks via **output variables**
- Runs as a generated AWS Lambda function behind an API Gateway

---

## Bot Lifecycle (API-First)

### Step 1: Create Empty Bot

```
POST /user/{userId}/bots
```

No body needed. Returns a new `IBot` object with:
- Generated `botId` (UUID)
- Generated `apiId` (API Gateway ID)
- Generated `triggerUrl` (the bot's HTTP endpoint)
- Empty `tasks: []`
- `active: false`

### Step 2: Configure Bot (Update)

```
PUT /user/{userId}/bots/{botId}
Body: { name, image, description, active, tasks }
```

This saves the bot configuration without deploying it. Use this for intermediate saves.

### Step 3: Deploy Bot

```
POST /user/{userId}/bots/{botId}/deploy
Body: { name, active, tasks }
```

This generates the Lambda code from the task definitions, packages it, and deploys it. If `active: true` and the trigger is a schedule, the scheduler is enabled.

### Step 4: Test Individual Tasks

```
POST /user/{userId}/bots/{botId}/test/{taskIndex}
Body: { task } (the task object at that index)
```

Tests a single task in isolation. Task 0 uses triggerSamples; tasks 1+ invoke the actual service Lambda.

## Data Models

### IBot (Top-Level)

```typescript
interface IBot {
  botId: string          // UUID, generated by backend
  userId: string         // Owner's user ID
  apiId: string          // API Gateway ID, generated by backend
  modelId?: string       // Reference to a bot model template (optional)
  name: string           // Display name (user-provided)
  description?: string   // What the bot does (user-provided)
  image?: string         // Icon URL (user-provided)
  active: boolean        // Whether the bot is currently deployed and running
  triggerUrl: string     // The HTTP endpoint to invoke this bot (generated)
  triggerSamples: ITaskExecutionResult[]  // Captured trigger invocations
  tasks: ITask[]         // The workflow steps (this is what you build)
}
```

### ITask (Workflow Step)

```typescript
interface ITask {
  taskId: number              // Unique ID, use Date.now() when creating
  app?: IApp                  // Which app this task belongs to
  service?: IService          // Which service to execute
  inputData: IVariable[]      // Input fields with values/mappings
  returnData?: boolean        // If true, this task's output is the bot's HTTP response
  sampleResult?: ITaskExecutionResult  // Cached test result (populated after testing)
  conditions?: ITaskCondition[][]      // Conditional execution (optional)
  connectionId?: string | number       // OAuth connection ID (for OAuth services)
}
```

### IService (What a Task Does)

```typescript
interface IService {
  type: 'trigger' | 'invoke'    // trigger = first task only; invoke = action tasks
  name: ServiceName             // Service identifier (see enum below)
  label: string                 // Human-readable name
  config: IServiceConfig        // Service-specific configuration
}

// Service names:
enum ServiceName {
  'code-execute'    // Run JavaScript code
  'http-request'    // Direct HTTP call
  'queue-publish'   // Publish to queue
  'oauth2-request'  // OAuth2 authenticated request
  'method-execute'  // Execute a predefined method
  'webhook'         // Webhook trigger (first task)
  'schedule'        // Schedule trigger (first task)
}
```

### IServiceConfig

```typescript
interface IServiceConfig {
  methodName?: MethodName       // For method-execute services
  customFields?: boolean        // Whether user can add custom input fields
  inputFields?: IVariable[]     // Template input fields (from app definition)
  outputPath?: string           // JSON path to extract from service response
  outputMapping?: object        // Field mapping for structured output
}

enum MethodName {
  'getTodo'           // Get user's todo list
  'publishToFeed'     // Publish content to the feed
  'sendNotification'  // Send push notification
  'httpRequest'       // Generic HTTP request
  'oauth2Request'     // OAuth2 authenticated request
}
```

### IApp (Service Provider)

```typescript
interface IApp {
  name: string           // App name (e.g., 'Baita', 'ChatGPT', 'NewsAPI')
  appId: string          // Unique app identifier
  config: IAppConfig     // App-level configuration
}

interface IAppConfig {
  apiUrl?: string        // Base API URL for the app
  authorizeUrl?: string  // OAuth authorization URL
  auth?: {               // OAuth token exchange config
    type: string
    method: string
    url: string
    headers?: Record<string, string>
    fields?: { username: string, password: string }
  }
}
```

### IVariable (Input/Output Field)

```typescript
interface IVariable {
  type: VariableType       // How this variable gets its value
  name: string             // Internal property path (e.g., 'bodyParams.messages.0.content')
  label: string            // Display label for the user
  value?: DataType         // Current/static value
  sampleValue?: DataType   // Value used during testing
  description?: string     // Help text
  required?: boolean       // Whether the field must have a value
  outputIndex?: number     // Which task's output to reference (0-based)
  outputPath?: string      // JSON path within that task's output
  customFieldId?: number   // For user-added custom fields
  options?: { label: string, value: string }[]  // For 'options' type
}

enum VariableType {
  'code'         // JavaScript code editor field
  'user'         // Auto-filled by user context (Token, TimeZone)
  'text'         // Free text input
  'output'       // References output from a previous task
  'options'      // Dropdown/select from predefined options
  'boolean'      // Toggle/checkbox
  'constant'     // Fixed value (not editable by user at runtime)
  'environment'  // References a server-side environment variable
}
```

### ITaskCondition (Conditional Execution)

```typescript
interface ITaskCondition {
  operator: ConditionOperator
  operand: IVariable           // Left-hand value to check
  comparisonOperand?: IVariable // Right-hand value (for comparison operators)
}

enum ConditionOperator {
  'Equals'
  'Not equals'
  'Exists'
  'Do not exists'
  'Contains'
  'Starts with'
  'Ends with'
}
```

Conditions are structured as `ITaskCondition[][]` where:
- Outer array = OR groups (any group passing means task runs)
- Inner array = AND conditions (all conditions in a group must pass)

## Variable Chaining Between Tasks

The key mechanism that makes bots powerful is **output variable chaining**: a task can reference the output of any previous task.

### How It Works

1. Task N produces an `outputData` object (captured in `sampleResult.outputData`)
2. Task N+1 (or any later task) can reference that output via an `IVariable` with:
   - `type: 'output'`
   - `outputIndex: N` (0-based index of the source task)
   - `outputPath: 'path.to.field'` (dot-notation path into the output object)
   - `name: 'targetField.path'` (where to place the value in this task's input)

### Example: Webhook → Code → ChatGPT

```
Task 0 (webhook) outputs: { prompt: "Hello world" }
Task 1 (code-execute) references: outputIndex=0, outputPath="prompt"
Task 1 outputs: { text: "processed: Hello world" }
Task 2 (ChatGPT) references: outputIndex=1, outputPath="text"
```

### The `name` Field Path Convention

The `name` field in `inputData` variables uses dot-notation to place values into nested objects:

- `bodyParams.messages.0.content` → `{ bodyParams: { messages: [{ content: VALUE }] } }`
- `queryParams.q` → `{ queryParams: { q: VALUE } }`
- `headers.Authorization` → `{ headers: { Authorization: VALUE } }`
- `notification.title` → `{ notification: { title: VALUE } }`

## Schedule Expressions

For schedule-triggered bots, the `expression` input supports AWS EventBridge syntax:

| Expression | Description |
|-----------|-------------|
| `rate(10 minutes)` | Every 10 minutes |
| `rate(30 minutes)` | Every 30 minutes |
| `rate(1 hour)` | Every hour |
| `cron(0 9 * * ? *)` | Daily at 9:00 AM |
| `cron(0 8,12,16,20 * * ? *)` | Daily at 8:00, 12:00, 16:00, 20:00 |
| `cron(0 9 ? * MON-FRI *)` | Weekdays at 9:00 AM |
| `cron(0 9-17 ? * MON-FRI *)` | Every hour Mon-Fri 9AM-5PM |

The `timeZone` field must also be provided (e.g., `Europe/Amsterdam`, `America/Sao_Paulo`).

## Important Rules for Building Valid Bots

### Task Ordering Rules

1. **Task 0 MUST be a trigger** — either `webhook` or `schedule`
2. **Tasks 1+ MUST be invoke services** — they cannot be triggers
3. **Tasks execute sequentially** — each task can reference output from any previous task
4. **Only one trigger per bot** — the first task defines how the bot starts

### Input Data Rules

1. **`inputData` on the task** is the actual values/mappings the user has configured
2. **`service.config.inputFields`** is the template defining what fields are available
3. For `type: 'output'` variables: `outputIndex` and `outputPath` are required
4. For `type: 'constant'` variables: `value` contains the fixed value
5. For `type: 'environment'` variables: `value` is the env var name (resolved server-side)
6. For `type: 'user'` variables: `value` is auto-filled at deploy time (Token, TimeZone)
7. For `type: 'options'` variables: `value` must match one of the `options[].value` entries
8. For `type: 'code'` variables: `value` is the JavaScript code string

### Code Execute Rules (Run JavaScript)

- The code MUST assign to the `output` variable to produce output
- Variables declared in `inputData` custom fields are available by their `name`
- The code runs in a sandboxed Lambda environment (Node.js 20.x)
- Access previous task outputs through the `inputEvent` variable (stringified JSON of trigger data)
- Maximum execution time: 300 seconds

### Output Referencing Rules

- `outputIndex` is 0-based (0 = trigger task, 1 = first invoke task, etc.)
- `outputPath` uses dot-notation: `field.nested.0.item`
- Empty `outputPath` (`""`) means the entire output object
- The `label` for output variables should show the sample value: `"fieldName: sample value"`
- The `sampleValue` field stores the last known output value for display purposes

### User Fields (Auto-Filled)

These are filled automatically at deploy time:
- **Token** — The user's push notification subscription token
- **Time Zone** — The user's browser timezone (e.g., `Europe/Amsterdam`)

### returnData Flag

- Set `returnData: true` on exactly ONE task to return its output as the bot's HTTP response
- Only meaningful for webhook-triggered bots
- If no task has `returnData: true`, the bot returns a generic success response

### Conditions

To add conditional execution to a task:
```json
"conditions": [
  [
    {
      "operator": "Equals",
      "operand": { "type": "output", "name": "status", "label": "status", "outputIndex": 1, "outputPath": "status" },
      "comparisonOperand": { "type": "text", "name": "value", "label": "success", "value": "success" }
    }
  ]
]
```

## Deployment Flow Summary

1. **Create** → `POST /user/{userId}/bots` (empty bot)
2. **Configure** → `PUT /user/{userId}/bots/{botId}` (add tasks, name, etc.)
3. **Test** → `POST /user/{userId}/bots/{botId}/test/{taskIndex}` (test each task)
4. **Deploy** → `POST /user/{userId}/bots/{botId}/deploy` (activate)
5. **Trigger** → POST to `triggerUrl` (for webhook bots) or wait for schedule

## Notes for AI-Assisted Creation

When helping a user create a bot:

1. **Ask about the trigger**: "Should this bot run on a schedule or be triggered by a webhook?"
2. **Understand the goal**: What should the bot accomplish? Map it to available services.
3. **Plan the data flow**: Which task produces what data, and how does it chain to the next?
4. **Use `Date.now()`** for taskId values (ensure each is unique)
5. **Include the full service config** including `inputFields` from the app definitions — the backend needs them for code generation
6. **Set `sampleValue`** on output variables to help the user understand what data flows where
7. **The `label` on output variables** should show context: `"fieldName: sample data"` format
8. **For scheduled bots**, always include both `expression` and `timeZone` in task 0 inputData
9. **For webhook bots**, suggest what the expected POST body shape should be
10. **Validate**: Ensure all required fields have values before suggesting deployment
